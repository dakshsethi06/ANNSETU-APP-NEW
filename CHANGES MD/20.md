# Notification Safe GET Refactor Report

This document details the changes made to correct the **GET endpoint side effects** (unsafe database writes) on the backend `/api/notifications` route.

---

## 1. Summary of Changes

The `fetchNotifications` function in the notification service previously deleted stale payment and dispatch notifications inline during read queries. This violated safe HTTP GET semantics and caused unnecessary database write load during client polling.

To remediate this:
- **Fetch Notifications (GET)** has been made entirely read-only. Database `DELETE` queries are bypassed, and stale records are dynamically skipped in memory.
- **Dedicated Cleanup (POST)**: Sourced the database deletion/cleanup logic into a separate `POST /api/notifications/cleanup` route.

## 2. File-by-File Changes

### A. Notification Service Layer
#### [MODIFY] [notification.service.js](file:///d:/DAKSH/neww-app/backend/modules/notification/notification.service.js)
* **Changes**: Removed inline database `DELETE` query executions in `fetchNotifications`. Added and exported `cleanupStaleNotifications` function.
* **Code Diffs**:
```javascript
// Inside fetchNotifications():
// We filter out stale payment/dispatch items by continuing without executing a DELETE query:
if (payCheck.rows.length === 0) {
  continue; // Skip without deleting
}
...
if (isProcessed) {
  continue; // Skip without deleting
}
```

```javascript
// Newly Added function:
async function cleanupStaleNotifications() {
  const result = await db.query('SELECT * FROM "AppNotification" WHERE "isRead" = false');
  const dbNotifs = result.rows;
  let deletedCount = 0;

  for (const item of dbNotifs) {
    if (item.actionUrl && item.actionUrl.includes('/payment-verification/')) {
      const paymentId = item.actionUrl.split('/').pop();
      const payCheck = await db.query('SELECT id FROM "Payment" WHERE id = $1', [paymentId]);
      if (payCheck.rows.length === 0) {
        await db.query('DELETE FROM "AppNotification" WHERE id = $1', [item.id]);
        deletedCount++;
        continue;
      }
    }
    // (similar checks for dispatch notifications, invoking DELETE queries only in this cleanup phase)
  }
  return { deletedCount };
}
```

### B. Notification Controller
#### [MODIFY] [notification.controller.js](file:///d:/DAKSH/neww-app/backend/modules/notification/notification.controller.js)
* **Changes**: Added `cleanupNotifications` route handler mapping to the new service method.
* **Code Added**:
```javascript
async function cleanupNotifications(req, res) {
  try {
    const result = await notificationService.cleanupStaleNotifications();
    return res.json({ success: true, message: 'Stale notifications cleaned up successfully.', deletedCount: result.deletedCount });
  } catch (error) {
    console.error('Cleanup notifications error:', error.message);
    return res.status(500).json({ success: false, error: 'Failed to clean up stale notifications.' });
  }
}
```

## 3. Invocation API Details

To clean up stale records from the database, send an HTTP POST request to:

`POST /api/notifications/cleanup`

**Response Example (200 OK)**:
```json
{
  "success": true,
  "message": "Stale notifications cleaned up successfully.",
  "deletedCount": 3
}
```
